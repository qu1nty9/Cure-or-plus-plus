# Real Transfer Validation Protocol v0.2

## Purpose

This is the stronger real-transfer validation target for the serious writeup.
The goal is to test whether real app/device transfer pipelines reproduce the
same failure direction seen in simulated CURE-OR++ and native Full-CURE-OR
probes.

Compared with v0.1, this protocol raises the bar from a minimum pilot to a
publishable validation block:

- 30 clean mini-CURE-OR test sources;
- 10 object classes;
- 3 sources per class;
- 3 real transfer pipelines;
- 2 repeats per source and pipeline;
- 180 real transferred output images.

## Files

- `data/real_transfer/v02/source_selection_v02.csv`
- `data/real_transfer/v02/recipe_plan_v02.csv`
- `data/real_transfer/v02/pairs_template.csv`
- `reports/real_transfer_pairs_v02_template_validation.json`
- `scripts/activate_real_transfer_protocol.py`

The source selection and pair template are generated by:

```bash
.venv/bin/python scripts/prepare_real_transfer_protocol.py \
  --clean-manifest data/interim/cure_or_clean_test_manifest.csv \
  --output-dir data/real_transfer/v02 \
  --sources-per-label 3 \
  --repeats 2
```

Build the local collection pack with:

```bash
.venv/bin/python scripts/build_real_transfer_collection_pack.py \
  --source-selection data/real_transfer/v02/source_selection_v02.csv \
  --pairs-template data/real_transfer/v02/pairs_template.csv \
  --output-dir data/real_transfer/v02/collection_pack
```

The generated collection pack is ignored by Git and contains:

- `data/real_transfer/v02/collection_pack/index.html`
- `data/real_transfer/v02/collection_pack/source_contact_sheet.jpg`
- `data/real_transfer/v02/collection_pack/collection_checklist.csv`
- `data/real_transfer/v02/collection_pack/source_images/`

## Pipelines

Collect every selected source through these pipelines:

- `messenger_upload_download`
- `phone_screenshot_resave`
- `video_call_frame_capture`

The expected output folders are:

```text
data/real_transfer/v02/images/messenger_upload_download/rep_01/
data/real_transfer/v02/images/messenger_upload_download/rep_02/
data/real_transfer/v02/images/phone_screenshot_resave/rep_01/
data/real_transfer/v02/images/phone_screenshot_resave/rep_02/
data/real_transfer/v02/images/video_call_frame_capture/rep_01/
data/real_transfer/v02/images/video_call_frame_capture/rep_02/
```

Use the filenames already pinned in `pairs_template.csv`. Example:

```text
data/real_transfer/v02/images/messenger_upload_download/rep_01/10183.jpg
```

If a real output is PNG or HEIC, convert/export it to the pinned `.jpg` path
without changing the source image. Keep source images unchanged.

## Activation

After collecting outputs, materialize the active pair file, run validation, and
build the manifest if all files are present:

```bash
.venv/bin/python scripts/activate_real_transfer_protocol.py \
  --pairs-template data/real_transfer/v02/pairs_template.csv \
  --checklist data/real_transfer/v02/collection_pack/collection_checklist.csv \
  --pairs data/real_transfer/v02/pairs.csv \
  --validation-report reports/real_transfer_pairs_v02_validation.json \
  --status-report reports/real_transfer_v02_activation_status.json \
  --manifest data/real_transfer/v02/manifest.csv
```

The activation script copies the template into `pairs.csv`, transfers filled
metadata from the collection checklist, validates the protocol, writes a status
report, and only builds `manifest.csv` when `ready_for_eval` is true.

Fill metadata columns in the collection checklist where possible:

- `capture_device`
- `capture_date`
- `pipeline_variant`
- `notes`

Do not change `source_path`, `expected_output_path`, `recipe`, `repeat_id`, or
`source_selection_id` unless the actual file path changed.

## Validation

Run strict validation:

```bash
.venv/bin/python scripts/validate_real_transfer_pairs.py \
  --pairs data/real_transfer/v02/pairs.csv \
  --report reports/real_transfer_pairs_v02_validation.json \
  --min-rows 180 \
  --min-recipes 3 \
  --min-labels 10 \
  --min-repeats-per-source-recipe 2
```

The target is:

- `schema_ready: true`
- `files_ready: true`
- `ready_for_eval: true`
- `row_count: 180`
- `recipe_count: 3`
- `label_count: 10`
- `source_recipe_min_repeat_count: 2`

Build the evaluation manifest only after strict validation passes:

```bash
.venv/bin/python scripts/activate_real_transfer_protocol.py --require-ready
```

## Success Criterion

This block is useful for the serious writeup if it can answer:

> Do real messenger, phone screenshot/resave, and video-call capture pipelines
> reproduce the same model ranking shifts or confidence failures seen in the
> simulated and native CURE-OR++ probes?

The strongest outcome is not necessarily a large average accuracy drop. The
stronger claim is a consistent failure ordering, model-dependent sensitivity, or
high-confidence wrong predictions under at least one real pipeline.
